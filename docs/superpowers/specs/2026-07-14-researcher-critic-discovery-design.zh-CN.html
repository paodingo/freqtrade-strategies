发现层设计

01 · PURPOSE

目的

本设计在既有 Research Director 之前增加一个受治理的研究发现层，填补当前系统的一处明确缺口：现有系统能够排序并治理已有、带证据链接的提案，却没有一个广覆盖、相互独立的机制来发现新的研究方向与完全不同的策略族。

发现层应让非专业人工审阅者能够作出知情的研究组合决策，而不要求其判断量化假设是否盈利。系统必须说明一个方向为何值得研究、可能错在哪里、成本是多少，以及哪些证据会叫停它。

获批流程为：

event trigger → Researcher → Research Critic → Top 3 → human direction review → Research Director → execution-risk approval → Campaign
方向批准的含义：仅表示某个想法值得进入正式准备；它不授权创建 Candidate、修改策略、访问受保护数据或执行 Campaign。
02 · GOALS

目标

  • 发现有证据链接、且不局限于对 RegimeAwareV6 渐进修改的研究方向。
  • 允许完全不同的策略族，同时固定初始市场、数据、运行时、时间周期与风险边界。
  • 每个发现周期最多输出三个用通俗语言解释并完成排序的方向。
  • 在人工审阅前强制进行独立的对抗性批评。
  • 保留 Research Director 的唯一权限：只有它能把获批方向转换为 research-proposal-v1 并应用正式治理。
  • 把 no_research_recommended 视为有效且预期的结果。
  • 保留完整来源追溯、确定性排序、不可变审批指纹与仅追加审计证据。
03 · NON-GOALS

非目标

发现层不执行以下操作：

  • 创建或编辑策略代码；创建 Candidate；运行 backtest、Campaign、hyperopt 或 live/dry-run 交易进程。
  • 访问 Validation 结果、Holdout 数据、私有 API、secret 或 live account 数据。
  • 获取并静默纳入新的市场数据集。
  • 改变 leverage、stake、stoploss、ROI、protections、position stacking、position adjustment 或其他风险语义。
  • 批准自身方向或授权执行；重开已关闭研究分支。
  • 取代 Research Director、Research Constitution 或既有审批路线。
  • 按承诺收益、win rate 或醒目的 backtest 表现给想法排序。
04 · SCOPE

固定的第一版范围

第一版可提出任意策略族，包括 trend following、mean reversion、breakout、volatility、regime switching、market microstructure 或其他可证伪的策略族，但必须固定以下比较包络：

  • exchange 与 market：Binance USD-M Futures；margin mode：isolated。
  • 仅使用获批的本地 Development 市场数据；仅在既有 manifest 允许目标分析时使用获批 BTC 与 ETH Development 数据集。
  • primary timeframe：1h；informative timeframe：4h。
  • 沿用既有不可变 runtime、execution、split 与 evaluation 契约。
  • 沿用既有风险控制与风险参数。
  • 不得增加 pair、timeframe、dataset，不得使用 Validation feedback 或 Holdout。

需要改变范围的想法可以被描述，但必须分类为 out_of_v1_scope，不得进入 shortlist。缺少数据集时只生成数据就绪建议，不得生成策略 Campaign。

05 · ROLES

角色与权限

5.1 Discovery Trigger Controller

Controller 仅在以下事件之一发生时启动发现周期：Campaign 完成、研究分支正式关闭、Research Director 返回 no_research_recommended，或人工明确请求发现周期。

每个 trigger 均包含 event identifier、event type、当前 research-state fingerprint、Constitution fingerprint、allowed-source policy version 与 creation time。trigger fingerprint 使重复投递保持幂等。

5.2 Researcher

Researcher 只读取获批来源，每周期生成 6–10 个 research-idea-v1 工件；可提出完全不同的策略族，但每周期同一策略族不得超过两个想法。

Researcher 负责想法生成与来源归属，不负责 critique、approval、proposal compilation 或 execution。某个版本一旦进入 critique 便不可修改；revision 必须生成新版本与新 fingerprint。

5.3 Research Critic

Research Critic 是独立审阅阶段，无权编辑 Researcher 工件。它独立检查来源支持、反向证据、重复、可证伪性、数据就绪度、leakage、overfitting、成本真实性、transaction-cost 敏感性与可信失败模式。

  • pass：可参与排序。
  • revise：在新的不可变想法版本获得新 critique 前不可参与排序。
  • reject：当前周期不合格。

同一周期每个想法最多允许一次 Researcher revision，以避免无边界 Researcher–Critic 循环。

5.4 Human Direction Reviewer

人工审阅者接收最多三个想法的中文审阅包；机器字段键与枚举值保持英文。审阅者判断方向是否值得正式准备，每周期最多选一个想法移交，这与 Constitution 的 one-Campaign portfolio budget 一致。其他想法标记为 rejected 或 deferred；deferred 不会在以后周期自动执行。

决策绑定 shortlist fingerprint、idea fingerprint、critique fingerprint、research-state fingerprint 与 Constitution fingerprint；任一变化都会使 approval 失效。

5.5 Research Director

Research Director 只接受一个已选择、经人工批准且 fingerprint 有效的 handoff；它将方向转换为既有 research-proposal-v1，并独立重做 closure、duplication、evidence、dataset、runtime、budget、contamination、risk 与 approval-route 检查。

Director 可以拒绝 handoff。拒绝对该 idea version 是终局，并作为以后周期的证据返回发现层，不触发自动改写或重试。人工方向批准也不满足之后 medium/high-risk 的 execution approval；既有 Constitution 路线保持权威，尤其是新 Candidate 或策略结构仍需明确执行授权。

06 · CONTRACTS

数据契约

6.1 research-trigger-v1

Required fields：

schema_version · trigger_id · event_type · event_ref · research_state_fingerprint · constitution_fingerprint · source_policy_version · created_at · trigger_fingerprint

6.2 research-idea-v1

Required fields：

schema_version · idea_id · idea_version · strategy_family · title · plain_language_summary_zh · falsifiable_hypothesis · proposed_market_mechanism · supporting_evidence · contradictory_evidence · source_refs · novelty_vs_existing_research · required_datasets · data_readiness · fixed_scope_confirmation · minimal_test_method · comparison_baseline · expected_information_gain · estimated_cost · risk_class · contamination_risk · falsification_conditions · stop_conditions · known_limitations · research_state_fingerprint · semantic_fingerprint

工件不得包含策略实现，也不得承诺预计收益。

6.3 research-critique-v1

Required fields：

schema_version · critique_id · idea_id · idea_semantic_fingerprint · verdict · source_verification · duplicate_research_check · falsifiability_assessment · data_readiness_assessment · leakage_and_overfit_risks · transaction_cost_challenge · strongest_counterevidence · alternative_explanations · fatal_objections · score_adjustments · ranking_inputs · critic_fingerprint

ranking_inputs 提供确定性排序策略使用的五个归一化分量。每个值必须由 critique 论证；shortlist builder 不得编造或推断缺失值。

6.4 research-shortlist-v1

Required fields：

schema_version · discovery_run_id · eligible_idea_count · ranking_policy_version · ranked_ideas · recommended_idea_id · recommendation · recommendation_reason_zh · research_state_fingerprint · shortlist_fingerprint

ranked_ideas 包含 0–3 个条目，recommended_idea_id 可为 null。空列表必须设置 recommended_idea_id: null 与 recommendation: no_research_recommended。

6.5 research-direction-approval-v1

Required fields：

schema_version · discovery_run_id · decision · selected_idea_id · selected_idea_fingerprint · selected_critique_fingerprint · shortlist_fingerprint · research_state_fingerprint · constitution_fingerprint · reviewer_type · decision_reason_zh · decided_at · approval_fingerprint

decision 取 approved_for_director_handoff、rejected 或 deferred。只有 rejected 或 deferred 可令三个 selection 字段为 null；仅 approved_for_director_handoff 允许 handoff，且要求三个值全部存在。

6.6 research-direction-handoff-v1

handoff 包含获批 idea 与 critique 引用、approval 引用、全部绑定 fingerprint、拟议 research_question，并声明 execution 未授权。它绝不包含 Candidate 或可执行 Campaign。

07 · STATE MACHINE

状态机

成功移交路径：

discovered → criticized → shortlisted → human_approved → handed_to_director → converted | director_rejected

从 shortlisted 出发，rejected 与 deferred 对当前周期均为终态，不能再迁移到 handed_to_director。

其他终态或旁路状态：

critic_rejected · revision_exhausted · out_of_v1_scope · insufficient_source_evidence · data_readiness_required · fingerprint_invalidated · no_research_recommended

任何状态迁移都不能直接抵达 Candidate 创建、Campaign 编译或执行。

08 · SOURCES

来源治理

8.1 来源类别

  • Class A：冻结的仓库数据集、已完成研究工件、Research Registry 记录、branch closure、获批治理工件与官方交易所文档。
  • Class B：同行评议论文、可信 preprint、教材与可信机构研究报告。
  • Class C：公开策略仓库、blog、forum、social media、video、ranking 与 commercial claim。

想法至少需要一个 Class A 或 Class B 来源才能通过 critique。Class C 可以启发想法，但不能支撑结论；仅有 Class C 的想法必须被拒绝。

8.2 外部来源记录

每条外部来源记录包含 canonical URL、source class 与 publisher type、retrieval timestamp、可用时的 publication timestamp、所支持或反驳的具体 claim、可用时的 content fingerprint 或 immutable retrieval handle、staleness assessment，以及 licensing 或 quotation constraints。

仓库只保存来源 metadata、必要时的简短合规摘录与 Researcher 自己的总结；不得把完整版权作品复制进仓库。

8.3 数据发现与数据获取

Researcher 可以说明某个想法需要 open interest、order book、liquidation、basis、其他 pair 或公开数据类型，但不得下载或把该数据当作已存在。此类想法成为 data_readiness_required，需先获得独立的人工数据获取范围批准，方可在以后考虑。

09 · RANKING

候选生成与排序

每周期从 6–10 个想法开始，同一策略族最多两个；仅 verdict 为 pass 的 critique 进入排序。每个计分分量归一化到 [0, 1]：

base_score =
    0.30 * expected_information_gain
  + 0.20 * falsifiability_and_mechanism_clarity
  + 0.20 * feasibility_with_existing_data
  + 0.15 * novelty_and_non_duplication
  + 0.15 * robustness_relevance

final_score = base_score
  - risk_penalty
  - cost_penalty
  - contamination_penalty
  - source_penalty
FactorValuePenalty or result
Risklow / medium / high / forbidden0.00 / 0.05 / 0.15 / reject
Costlow / medium / high0.00 / 0.03 / 0.08
Contaminationnone / low / medium / high0.00 / 0.02 / 0.08 / reject
Sourcesincludes Class A / Class B without A / Class C only0.00 / 0.02 / reject

scoring policy 有版本且在一次 discovery run 内不可更改。想法必须满足 final_score >= 0.55 才能进入 shortlist。平分时依次按 lower risk、lower cost、semantic_fingerprint 的 lexical order 决胜，使相同输入得到可复现排序。

Expected return、backtest profit 与 win rate 不参与计分。输出最多三个想法；若无人越过 threshold，周期以 no_research_recommended 正常完成。

10 · HUMAN REVIEW

人工审阅包

面向人工的中文优先审阅包，会为每个 shortlist 想法说明：研究内容、为何当下值得关注、可能奏效的机制、最强反对理由、已有与缺少的数据、最小有效实验、时间/计算/治理成本、停止方向的结果、Critic verdict 与 score adjustments，以及系统建议与不确定性。

审阅包必须明确说明：批准一个方向不等于盈利判断，也不是执行授权。

运行期的 human-review Markdown、final audit Markdown 与 decision explanation 必须使用简体中文；checked-in technical design 与 implementation-plan Markdown 可保留英文，但每份都必须在同目录附完整、自包含的简体中文 .zh-CN.html 阅读副本。Markdown 与机器可读工件保持权威；HTML 必须离线可用、适配桌面与移动宽度、可干净打印、展示来源追溯，并以英文保留 code、path、command、schema name、key、enum 与精确 policy value。

11 · FAILURES

失败处理

  • 来源证据缺失或无法验证 → insufficient_source_evidence。
  • 所需数据缺失 → data_readiness_required，且不得创建策略 Campaign。
  • 重复研究需引用既有证据后拒绝；缺少 Critic 工件会阻断 shortlist。
  • source、idea、critique、state、Constitution 或 shortlist fingerprint 不匹配会使 approval 失效。
  • 外部获取不可用时，可仅凭仓库来源继续，并记录 source_coverage: internal_only_degraded；不得伪称完成外部审阅。
  • Research Director rejection 作为该 idea version 的终局证据，不触发自动重试。
  • 同一 trigger fingerprint 重复投递时返回既有 run，不生成新想法。
  • 没有合格想法时以 no_research_recommended 正常完成。
  • 任何访问 forbidden path、Validation、Holdout、private API、secret、strategy mutation path、Candidate path 或 execution runner 的尝试，都必须停止并升级该 run。

13 · VERIFICATION

验证与验收

13.1 Contract Tests

  • 六个 schema 接受完整有效工件并拒绝缺少 required field 的工件。
  • 未知或非法状态迁移 fail closed；fingerprint 对归一化相同内容稳定，并在 governed content 变化时改变。
  • approval 验证全部绑定 fingerprint；handoff 只接受 approved_for_director_handoff 并声明 execution_authorized: false。

13.2 Role-boundary Tests

  • Researcher code 不能调用 experiment、Campaign、Candidate 或 strategy mutation path；Critic code 不能编辑或替换 idea。
  • 二者均不得访问 Validation、Holdout、secret、private API 或 live data；新策略族可接受，超出固定 market/timeframe/dataset/risk 范围的变更被阻断。

13.3 Ranking Tests

  • 相同输入产生相同 score、tie-break 与 ordering；同一策略族初始想法不超过两个；shortlist 不超过三个。
  • Class C-only、forbidden-risk、high-contamination、duplicate 与 below-threshold idea 被排除；空 eligible set 产生 no_research_recommended。

13.4 Workflow Tests

  • Campaign completion、branch closure、Director exhaustion 与 manual request 均产生有效 trigger，且相同 trigger fingerprint 保持幂等。
  • Critic revise 要求新版本且每周期最多一次 revision；人工最多选择一个 handoff。
  • approval 后 content/state 变化使其失效；Director rejection 留证且不自动重试；handoff 不绕过 Director checks。

13.5 End-to-end Dry Run

验收必须证明：生成跨多个策略族的 6–10 个想法；独立 critiques；确定性 Top 3 或 no_research_recommended；可读中文 Markdown 与自包含 .zh-CN.html；fingerprint-bound 人工决策；无 execution authorization 的 Director handoff；Candidate 创建、Campaign 启动、Validation/Holdout 访问、strategy/risk mutation 均为 0；并具备完整 registry/artifact-integrity、targeted tests、readiness、baseline、guard、logical commit 与 clean versioned worktree 证据。

14 · SUCCESS

成功标准

当非专业人工能够从最多三个解释清晰、经过对抗审阅的方向中判断是否有一个值得正式准备，同时仓库能证明发现过程本身未实现策略、未运行实验，且既有 Research Director 与 Constitution 始终保有完整执行权时，发现层即视为成功。
