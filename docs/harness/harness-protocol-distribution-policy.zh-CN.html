Harness Protocol 静态分发政策

P4A 把 P1 Protocol Core 与 P2 Project Mapping 固化为可重复核验的静态分发描述，但不执行发布或消费者迁移。

批准边界：static_distribution_only。不包含 shared Runtime、CLI、package、plugin、skill、Role Pack、网络操作、sibling repo 写入、项目业务执行或 Campaign 执行，也不允许 publish 或 consumer rollout。

权威来源与内容

v0.x 的 canonical owner 是 https://github.com/paodingo/freqtrade-strategies.git 中的精确 commit。distribution v0.1 的 release-manifest.json 将冻结的 15 个输入文件绑定到 6363b7f8352a53cbcd709a4d3d6b5c0bc7ba3b93。该 commit 标识 canonical input snapshot，不自我引用后来加入生成 manifest 的实现 commit。

manifest 包含两个有序组件：

  1. protocol-core：P1 schema、P1 manifest 和 5 个合成 fixture。
  2. project-mappings：P2 schema、P2 manifest、3 个冻结项目 mapping descriptor 和 3 个合成失败 fixture。

每个 artifact 都记录 repo-relative POSIX path、component、media type、规范化 byte count 和 fingerprint。增、删或重排任一 artifact 都会产生不同 manifest，并需要新的显式批准。

跨平台指纹

sha256-text-lf-v1 是唯一支持的 profile：输入必须是无 BOM 的 UTF-8；非法 UTF-8、重复 JSON key 或未知 profile 均 fail closed；CRLF 与 lone CR 统一为 LF，其他内容保持不变；对规范化后的 UTF-8 bytes 计算 SHA-256。

P4A 之前产生的 P1/P2 fingerprint 保留原有的历史与 project-local 语义，P4A 不会静默重解释或改写它们。

源仓库构建与核验

.\.venv-freqtrade\Scripts\python.exe scripts/build_harness_protocol_distribution.py --check
.\.venv-freqtrade\Scripts\python.exe scripts/verify_harness_protocol_distribution.py

builder 只读取冻结的 15 个文件及本地 fingerprint profile；不带 --check 时也只写 exact release-manifest.json。verifier 核验 closed schema、deterministic reconstruction、P1/P2 membership relationship 与 source-side-only imports。两者都不会复制到消费者，也不会调用网络、项目 Runtime 或 sibling repo。

  • 0：passed。
  • 1：合同不匹配，blocked。
  • 2：工具或 parser 无法给出可信结论，error。

升级与回滚

不允许 auto-update。source commit、artifact fingerprint 或 component membership 任一变化都会使当前批准失效；新候选必须重新获得项目显式批准。

未来消费者只能显式 pin 到先前批准的 release 来回滚，同时保留自己的 project-local Runtime。自动清理、覆盖和迁移均不属于 P4A。

Gate B 与停止条件

Gate B 需要 exact 12-path diff、15-artifact deterministic manifest 与指纹、P1+P2+P4A focused tests、readiness、clean-worktree portable baseline、protected trading/business diff 为 0、sibling repos 不变，以及 fresh detached worktree 的独立审查。

即使 Gate B 通过，也不代表已发布。本阶段不会 tag、release、copy、install 或 rollout；这些动作需要另一份计划和新的显式批准。
