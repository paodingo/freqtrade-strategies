01 · PURPOSE

目的与核心决策

本设计从三个成熟度不同的真实项目中定义一个语言无关的 Harness Protocol。共享的是治理词汇、契约和 conformance 边界，而不是强制共享运行时、角色清单、存储引擎或领域模型。

核心决策：现在抽象协议，通过 Project Adapter 验证；至少两个项目证明相同语义后，才决定是否抽取共享可执行运行时。

规格暂时孵化在 freqtrade-strategies-clean，因为该仓库具有目前最完整的 Harness 生命周期。这不意味着 Freqtrade 领域模型对另外两个项目具有权威。

02 · AUTHORITY

授权边界

当前只授权文档。本设计不授权创建共享 package/CLI/repository，不授权启动 ChinaSectorRadar H1，不要求 rehab-intervention 引入 multi-agent framework，也不授权重构 Freqtrade 的 discovery、Director、Campaign、registry 或 execution。

后续任何实现都必须服从被修改项目自己的 Phase、Constitution、Task、approval、exact-path 和人工授权。共享协议只能收窄权限，不能扩大权限。

04 · GOALS

目标与非目标

目标

  • 用 JSON Schema、JSON、Markdown evidence 与 process exit semantics 表达语言无关协议。
  • 把 Harness governance、Role Pack、Project Adapter 和 Executor 分离。
  • 同时容纳 model-backed role、deterministic process 与 human decision。
  • 支持 Python、PowerShell、Node.js 项目而不提前选择共享运行时。
  • 保留 exact permission、explicit approval、fail-closed、baseline debt、provenance 与 audit evidence。

非目标

  • 不定义统一 BaseAgent；不要求所有项目使用 Researcher、Critic、Director 或 Campaign。
  • 不标准化业务状态、数据模型、评分、部署或风险语义。
  • 不替代 AGENTS.md、Constitution、Phase contract、quality baseline 或 project approval。
  • 不假设 Harness 成功就代表业务健康。
  • 不授权 production、live trading、market-data acquisition、database mutation、deployment 或 secret access。
05 · PRINCIPLES

设计原则

Contract coupling 是必要的

Agent 应耦合到稳定的 permission、input、output、approval 和 escalation contract，而不是任意路径、隐式聊天历史或可变约定。

Composition over inheritance

RoleContract
+ ProjectBinding
+ TaskManifest
+ CapabilityPolicy
+ Executor
= AgentRun

Protocol before runtime

先建立协议与 conformance suite。共享 CLI/library 的语言、包形态和所有权等 Adapter 证据充分后再决定。

Fail closed without erasing uncertainty

未知权限、无效 schema、缺少证据、stale binding、illegal transition、tool unavailable 或 authority ambiguous 都不能静默通过。blocked 与 error 必须区分。

Evidence is part of the result

completion claim 必须绑定 authority snapshot、command outcome、artifact identity、known blocker 与 final state。Harness completion 和 business readiness 是两个独立字段。

06 · ARCHITECTURE

分层架构

Cross-Project Harness Protocol ├── Core Contracts ├── Role Packs ├── Project Adapters └── Executors

Core Contracts 只表达 authority、scope、budget、gate、state、approval、escalation 与 evidence。Role Packs 是可选 workflow module。Project Adapters 映射现有术语与证据。第一版 Executors 继续留在项目本地。

Core 中不得出现项目名、trading pair、business entity、framework path、dataset id、scheduler name、database table、strategy family 或 UI concept。

07 · CONTRACTS

Core Contracts

ProjectBindingPhaseAuthorityRoleContractTaskManifestCapabilityPolicyBudgetGateResultRunStateApprovalRecordEscalationRecordEvidenceBundle

ProjectBinding 与 PhaseAuthority

ProjectBinding 定位 authority、phase、policy、runtime entry、state backend 和 evidence root，但本身不授予权限。PhaseAuthority 记录 approved/forbidden operation、protected surface、next phase 与 human approval requirement。Phase 变化默认使旧 Task authorization 失效。

RoleContract

定义 accepted input、produced output、allowed/forbidden capability、approval authority、self-approval 与 escalation。self_approval_allowed 默认是 false。model、deterministic code 或 human 都不能因实现类型而扩大角色权限。

TaskManifest

绑定 project、phase、role、objective、input、allowed/blocked paths、capability、budget、validation、expected output、stop condition、required approval 与 authority fingerprint。blocked_paths 永远优先于 allowed_paths。

CapabilityPolicy

Capability 使用 namespace，例如：

core.read_repository
core.write_allowlisted_artifact
core.run_validation
research.propose_direction
market.acquire_data
ops.run_scheduler
database.write
deploy.publish

Core 定义机制，不定义完整全局 capability 清单。Project Adapter 负责项目 namespace 和具体 tool/path binding。

GateResult

outcome含义portable exit
passedgate 完成且条件成立0
blockedgate 完成并发现治理违规或未满足条件1
error工具、环境或配置导致无法可信判断2

RunState

planned → ready → running → completed
                         ↘ stopped
                         ↘ failed
                         ↘ escalated

completed 仅表示授权任务正常结束，不表示业务 readiness。local state 和 local reason code 必须保留。

ApprovalRecord 与 EscalationRecord

Approval 绑定 action、artifact、fingerprint、scope 与时间；content、authority、phase 或 governed state 变化默认使其失效。Escalation 记录 reason、blocked action、required authority、evidence 与 safe resume condition，但不授予权限、不自动重试。

EvidenceBundle

必须包含 run/task identity、authority snapshot、input identities、gate/command results、artifact refs、known baseline debt、open blockers、business_readiness、harness_completion 与 final state。

08 · ROLE PACKS

可选 Role Packs

research-discovery

Researcher、ResearchCritic、HumanDirectionReviewer、ResearchDirector、CampaignOrchestrator、Evaluator、Gatekeeper。市场、dataset、contamination、Candidate、Validation 与 Holdout 仍属于 Freqtrade Adapter。

audit-and-review

Coordinator、DataResearcher、Developer、SpecReviewer、QualityReviewer、IndependentAuditor。Reviewer 不编辑自己批准的工件；本 pack 不自动创建 Product Manager。

implementation-and-review

Implementer、Reviewer、Verifier。普通应用项目可使用这个最小 pack，无须采用 research-discovery。

09 · ADAPTERS

三个 Project Adapter

Freqtrade

Constitution → authority/policy；Researcher/Critic/Director → RoleContract；Campaign YAML → TaskManifest + execution extension；Director Registry → state/evidence；fingerprints/approvals → ApprovalRecord；Validation/Holdout/Candidate → domain capability extension。

ChinaSectorRadar

AGENTS.md 与 H0 scope → PhaseAuthority；exact paths/protected surface → TaskManifest；Coordinator/DataResearcher/reviewer → audit Role Pack；H0 audit → EvidenceBundle；H1 task manifest/readiness → future implementation。该 mapping 不授权 H1。

rehab-intervention

Phase 2 Campaign JSON → early TaskManifest；Node runner → local Executor；allowlist/changed paths → path policy；attempt/file/time limit → Budget；validation → GateResult；state JSON → local RunState backend；auth/session/database/deploy → project capability extension。该 mapping 不增加产品 multi-agent framework。

10 · ADMISSION

二取三准入规则

  1. 至少两个项目需要实质相同的语义。
  2. 第三个项目不与该抽象冲突。
  3. 行为能够在不导入领域概念的前提下定义和测试。
  4. 共享层保留更严格的 project-local policy。
  5. 抽象具有明确失败行为。

11 · FLOW & FAILURES

协议数据流与失败处理

Project authority sources ↓ Project Adapter → canonical snapshot ↓ Protocol conformance validation ↓ Project-local Executor ↓ GateResult + local reason + artifacts ↓ RunState → EvidenceBundle ↓ complete / stop / fail / escalate
  • invalid protocol、tool/config 无法判断 → error。
  • authority mismatch、path violation、protected surface → blocked。
  • secret/private credential request → escalated，证据不复制 secret。
  • budget exhaustion → stopped，保留消耗证据。
  • 正确 Harness 发现业务 blocker 时，Harness 可以 completed，但 business readiness 继续 blocked/degraded。
  • approval fingerprint mismatch → invalidated，不自动重新批准。
  • retry 不扩大 scope、不刷新 budget、不改 policy、不跨 phase。
12 · CONFORMANCE

Conformance 模型

Contract

Schema 必须接受完整有效工件并拒绝缺失、未知 critical field 和非法 enum；fingerprint 对归一化相同内容稳定；authority/content 变化使依赖 binding 失效；blocked path 优先。

State 与 Gate

相同 local outcome 映射到相同 normalized result；非法 transition 不改变 state；resume 只从显式 resumable state 继续，且不补回 budget。

Evidence

证据必须包含 authority、task、commands、outputs、blockers、final state；Harness completion 与 business readiness 独立；baseline debt 保留；敏感发现只记录位置与风险。

Role boundary

Role 不能调用未授权 capability；Reviewer 不能静默编辑自己批准的工件；禁止 self-approval 的 role 不能为自身输出创建有效 approval。

Golden fixtures

每个 Adapter 至少提供 normal、governed block、tool/environment error、authority mismatch、known-baseline-debt 五类 synthetic 或 approved non-secret fixture。

13 · ADOPTION

分阶段采用

  1. P0 Design：仅本设计及 HTML，不改 runtime。
  2. P1 Schemas：另行批准后创建语言无关 schema 与 synthetic fixtures，不导入三个项目 runtime。
  3. P2 Read-only mappings：每项目一个 Adapter mapping，证明不会弱化 local rule，不替换 runner。
  4. P3 Second-consumer pilot：在一个低成熟度项目的下一次独立获批 Harness phase 中试用 protocol。
  5. P4 Runtime decision：至少两个 Adapter conformance、两个真实 governed run、ownership/distribution/rollback 明确后，才讨论 package、CLI、plugin 或 repository。
P3 候选可能是另行批准的 ChinaSectorRadar H1，也可能是未来 rehab-intervention Campaign。本设计不选择候选，也不授予任何执行权限。
14 · DEFERRED

有意延后的决定

以下并非遗漏，而是等待 Adapter 证据：shared CLI/library 的语言、共享仓库所有权、package/distribution、state storage、model provider interface、approval fingerprint 是否对所有低风险 Task 强制、Role Pack 是否随 Core 发布、Codex skill/plugin 是否作为分发面。

未来 implementation plan 不得静默决定这些事项；选择任何一项都需要明确的设计修订或 implementation-specific design。

15 · SUCCESS

成功标准与下一门禁

  • 三个项目都能映射真实能力，不伪装成拥有不存在的能力。
  • Core 不含任何 Freqtrade、A 股、rehab、framework、database、scheduler、strategy 或 UI 领域字面量。
  • 更严格的项目规则始终覆盖共享默认值。
  • Agent 是可测试 RoleContract，而不是隐藏的项目权限。
  • 业务 readiness 失败可以与正确 Harness completion 共存。
  • 没有共享 runtime 时也能用 synthetic fixtures 验证 conformance。
  • 第二消费者无需广泛重写第一项目。
当前批准方向：protocol_first。下一项允许提出的设计工作是 P1 schema 与 synthetic conformance fixture 规格；本文件本身不授权 P1 实现。
