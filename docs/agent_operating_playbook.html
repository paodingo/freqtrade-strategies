一句话结论

以后这个项目里的 AI 不能只凭感觉改策略、改前端、写报告。它必须先确认服务器真实状态， 再按测试、回测、健康检查和风险红线来推进。

服务器优先

本地报告只是开发证据，最终以线上服务器、容器、API 和健康检查为准。

策略不乱换

V11.16 是当前候选，V10.8.2 是赚钱基准。没有通过验收前不能随便替换。

AI 必须遵守的交易红线

不能为了让回测好看而破坏真实性。 这包括提高杠杆、放大单笔风险、删除手续费、忽略滑点、忽略资金费、使用未来数据。
  • OI（Open Interest，未平仓量）只能作为加分项。 缺失时记为中性，不能因为缺 OI 就禁止交易。
  • 不能伪造历史数据。 Binance 官方不能补齐的历史 OI 缺口，必须报告缺口。
  • 新策略必须隔离。 新 class、新 config、新数据库、新端口、新报告目录，避免污染线上基准。
  • 失败版本要留下证据。 失败报告不能删，因为它能防止反复走同一条错路。

以后 AI 改东西的标准流程

  1. 先读 AGENTS.md 和相关代码、脚本、测试、报告。
  2. 判断这次改动是策略、前端、部署、告警、数据，还是纯文档。
  3. 能写测试的行为改动，先写或更新测试。
  4. 本地验证通过后，如果涉及线上展示或模拟盘，再同步到服务器验证。
  5. 最后回答时必须说明：改了什么、验证了什么、服务器状态如何、还有什么风险。

告警排查不能混为一谈

Telegram 报 jq parse error、API 异常、stopped，并不一定代表策略真的停止交易。 AI 必须分层检查：

先排运行链路

  • 告警脚本是不是 JSON 拼坏了？
  • Dashboard API 有没有 500？
  • Docker 容器是不是还在？
  • Freqtrade API 的 state 是不是 running？

再排交易逻辑

  • 有没有 pairlock？
  • 是不是策略过滤器主动不交易？
  • 最近持仓、成本、滑点是否异常？
  • 亏损来自策略问题还是市场窗口变化？

Skills 够不够？

够用。真正的问题不是 AI 缺工具，而是缺项目级纪律。以后按下面这个路由来用：

情况 应该使用的能力 通俗解释
改策略或行为逻辑 Test-driven development 先用测试锁住预期，再改代码。
遇到告警、接口错、状态不一致 Systematic debugging 按证据分层排查，不凭感觉猜。
大改架构或阶段目标 Writing plans 先写方案和验收标准，再执行。
改前端展示 Frontend testing/debugging 要看页面、查接口、验证展示是否和真实策略一致。
准备说完成 Verification before completion 必须先跑验证命令，再给结论。

以后文档的默认交付格式

默认两份： Markdown 给 AI 和工程流程使用；HTML 给你阅读、复盘、截图和快速判断。
  • HTML 使用中文写，尽量少堆英文。
  • 必须出现英文专业词时，要给中文解释，比如 Profit Factor（盈利因子）。
  • 报告类 HTML 要突出结论、是否达标、失败原因、下一步，而不是只堆表格。
  • 操作手册类 HTML 要写清楚“遇到什么情况，按什么顺序做”。

最终交付时必须说清楚

已完成：改了哪些文件，解决什么问题
验证：跑了哪些命令，结果是什么
服务器状态：如果涉及线上，说明容器/API/健康检查状态
剩余风险：还有哪些没有验证，或者需要继续观察

这能避免 AI 给出“看起来完成了”的回答，也方便你快速判断这一步到底靠不靠谱。
